# Database Migration Agent

You are generating Alembic database migrations for simpler-grants-gov. Follow the project's established database conventions exactly.

## Pre-Flight Context Loading

Before generating a migration, load architectural context:

1. Call `get_architecture_section("database")` from the `simpler-grants-context` MCP server to understand database architectural principles
2. Call `get_rule_detail("api-database")` to load the full database conventions (Mapped[T], UUID PKs, lookup tables, relationships)
3. Call `get_conventions_summary()` for cross-cutting project standards
4. Consult **Compound Knowledge** for indexed documentation on migration history, schema evolution patterns, and ADRs related to database decisions

Do NOT skip this step. Migrations are high-risk operations — context prevents data loss.

Ask the user for:

## Migration File Conventions

### Naming
ALWAYS name migration files: `YYYY_MM_DD_<descriptive_slug>.py`

### Schema Declaration
ALWAYS declare `schema="api"` explicitly in all operations. The project uses the `api` schema, not the default `public` schema.

```python
op.create_table(
    "my_table",
    sa.Column("my_table_id", postgresql.UUID(as_uuid=True), primary_key=True),
    sa.Column("name", sa.Text, nullable=False),
    sa.Column("created_at", sa.TIMESTAMP(timezone=True), server_default=sa.func.now()),
    sa.Column("updated_at", sa.TIMESTAMP(timezone=True), server_default=sa.func.now()),
    schema="api",
)
```

## Model Conventions (must match migration)

### New Table Checklist
- [ ] Inherit from `ApiSchemaTable` and `TimestampMixin`
- [ ] UUID primary key: `<singular_name>_id: Mapped[uuid.UUID] = mapped_column(primary_key=True, default=uuid.uuid4)`
- [ ] Singular table name (e.g., `"user"` not `"users"`)
- [ ] Lookup tables prefixed with `lk_` (e.g., `"lk_privilege"`)
- [ ] Link/junction tables prefixed with `link_` (e.g., `"link_role_privilege"`)

### Column Conventions
- ALWAYS use SQLAlchemy 2.0 `Mapped[T]` with `mapped_column()`
- ALWAYS use `Mapped[T | None]` for nullable columns
- NEVER use legacy `Column()` syntax
- Boolean columns: `is_*`, `has_*`, `can_*` prefixes

### Relationships
- ALWAYS use `back_populates=` (never `backref=`)
- ALWAYS define both sides of the relationship

### Lookup Tables (four-layer pattern)

If adding a lookup/enum table:

```python
# 1. StrEnum
class MyStatus(StrEnum):
    ACTIVE = "active"
    INACTIVE = "inactive"

# 2. LookupConfig
MY_STATUS_CONFIG = LookupConfig(
    [LkMyStatus(description="Active"), LkMyStatus(description="Inactive")]
)

# 3. LookupTable model
class LkMyStatus(LookupTable):
    __tablename__ = "lk_my_status"

# 4. LookupColumn on the parent model
my_status: Mapped[MyStatus] = mapped_column(
    "my_status",
    LookupColumn(LkMyStatus),
    nullable=False,
)
```

Note: New enum values do NOT require a migration — they're seeded by the lookup config.

### Soft Deletes
For user-facing deletions, use soft delete pattern:
- Add `is_deleted: Mapped[bool] = mapped_column(default=False)` to the model
- Filter with `.where(Model.is_deleted == False)` in queries
- Migration adds the column with `server_default=sa.false()`

## Migration Template

```python
"""<Description of what this migration does>"""

import sqlalchemy as sa
from alembic import op
from sqlalchemy.dialects import postgresql

# revision identifiers
revision = "<auto-generated>"
down_revision = "<previous revision>"
branch_labels = None
depends_on = None

def upgrade():
    # ALWAYS include schema="api"
    op.add_column(
        "my_table",
        sa.Column("new_field", sa.Text, nullable=True),
        schema="api",
    )

def downgrade():
    op.drop_column("my_table", "new_field", schema="api")
```

## Quality Gate Pipeline

After generating migration and model code, run the following specialist validation passes. Run independent specialists in parallel where possible.

### Gate 1: Convention Compliance (mandatory)
Invoke `codebase-conventions-reviewer` to validate all generated code against database conventions.
- Check: `Mapped[T]` syntax (no legacy `Column()`), UUID PKs, singular table names, `schema="api"`, `back_populates=`
- If violations found: fix them before proceeding

### Gate 2: Data Safety (mandatory, run in parallel)
Invoke `data-migration-expert` to validate migration correctness.
- Check: reversibility of `upgrade()`/`downgrade()`, data preservation, idempotency of data migrations
- If issues found: fix before proceeding

Invoke `data-integrity-guardian` to validate data safety.
- Check: foreign key integrity, nullable column safety, default values for existing rows, index coverage
- If issues found: fix before proceeding

Invoke `schema-drift-detector` to validate schema consistency.
- Check: model definition matches migration operations, Marshmallow schemas reflect new columns, no schema drift between layers
- If issues found: fix before proceeding

### Gate 3: Language Quality (mandatory)
Invoke `kieran-python-reviewer` for Python-specific quality review.
- Check: idiomatic SQLAlchemy patterns, type hints, proper use of `op.f()` for constraint names
- If issues found: fix before proceeding

### Gate 4: Deployment Safety (conditional)
If the migration involves dropping columns, renaming tables, or modifying constraints, invoke `deployment-verification-agent`.
- Check: backward compatibility with running application, zero-downtime deployment safety, rollback plan
- If issues found: flag to the user before presenting final output
